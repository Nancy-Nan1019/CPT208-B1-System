# PRD - XJTLU 课堂讨论平台

## 1. 产品概述

### 1.1 背景
CPT208 Human-Centric Computing 课程项目，话题 B1 XJTLU Group Discussions。  
针对课堂小组讨论中参与度不均衡、内向学生不愿开口、教师难以监控全局的问题，设计一款游戏化讨论辅助 Web App。

### 1.2 目标
- 提升小组讨论中每位成员的参与积极性
- 通过可视化手段让参与度不均衡问题显性化
- 利用 GenAI 在讨论卡顿时提供智能引导

### 1.3 用户角色

| 角色 | 描述 |
|---|---|
| Teacher（教师） | 设置讨论主题、触发分组、查看全局数据 |
| Student（学生） | 参与讨论、操作 AI 小人、查看积分排行 |

---

## 2. 功能需求

### 2.1 公共模块

#### 2.1.1 登录 / 注册
- 用户通过邮箱 + 密码注册账号
- 注册时选择角色：Teacher / Student
- 登录后根据角色跳转对应首页
- 支持记住登录状态（7天免登录）

---

### 2.2 学生端

#### 2.2.1 性格选择（首次登录）
- 学生首次登录后进入性格选择页
- 提供 E（外向 Extrovert）/ I（内向 Introvert）两个选项
- 选择后保存至个人档案，后续可在设置中修改

#### 2.2.2 等待室
- 学生登录后进入等待室，显示当前 Session 信息
- 实时显示已加入人数
- 等待教师触发分组
- 分组完成后自动跳转至讨论室

> **[难点]** 分组完成后的状态广播需要 WebSocket/Realtime 推送，所有学生端必须同时收到跳转指令，需处理网络延迟和断线重连。

#### 2.2.3 讨论室
**AI 小人区（核心功能）**
- 页面中央展示本组所有成员的 AI 小人
- 每个小人对应一名学生，显示其昵称
- 小人状态：
  - 静止：该学生未发言
  - 跑动动画：该学生正在发言（按住发言按钮期间）
  - 停止：发言结束
- 小人跑动速度与发言时长成正比（发言越多跑越快）

> **[难点 ⭐⭐⭐⭐]** 多个小人动画状态需要与实时同步数据联动，每个小人的跑动速度动态变化，动画帧与网络状态之间需要做平滑插值处理，避免卡顿跳变。

**发言按钮**
- 页面底部显示「按住发言」按钮
- 按住期间：本人小人跑动，积分实时累加，状态广播给组内所有人
- 松开后：小人停止，积分结算

> **[难点 ⭐⭐⭐]** 按住状态需要以高频率（建议 200ms 一次）向服务端发送心跳，服务端广播给组内其他人。需处理移动端长按事件（touchstart/touchend）和 PC 端（mousedown/mouseup）的兼容，以及意外断线时的状态回滚。

**AI 引导气泡**
- 触发条件：
  - 某学生沉默超过 60 秒 → AI 小人跑至该学生小人旁，气泡提示个性化问题
  - 全组沉默超过 30 秒 → AI 小人居中，气泡提出追问推进讨论
  - 讨论时间达到 80% → AI 小人提示进入总结阶段
- 引导内容由 DeepSeek/OpenAI API 根据当前讨论主题和沉默情况动态生成

> **[难点 ⭐⭐⭐⭐⭐]** 最高复杂度模块。需要：① 服务端维护每个学生的沉默计时器；② 将讨论主题 + 沉默人员 + 已发言内容摘要组装成有效 Prompt；③ 调用 AI API 并处理响应延迟（需 loading 状态）；④ AI 小人跑动位移动画需配合气泡出现时机；⑤ 同一时间只能触发一条引导，需加队列防止并发重复触发。

**实时积分排行榜（右侧）**
- 实时显示本组所有成员当前积分
- 积分规则：发言时长（秒）× 基础系数
- 排名变化时有动画效果
- 讨论结束后展示最终排名

> **[难点 ⭐⭐⭐]** 排行榜需要订阅实时数据流，每次积分变化都要触发列表重排序动画（AnimatePresence/layout animation），需防止高频更新导致的动画闪烁。

**讨论进度条**
- 顶部显示本次讨论剩余时间
- 类似跑步闯关的赛道进度可视化

> **[难点 ⭐⭐]** 倒计时需以服务端时间为准（而非客户端本地时间），避免各端时间不一致。

---

### 2.3 教师端

#### 2.3.1 Session 管理
- 创建新的讨论 Session，填写：
  - 讨论主题（传入 AI 作为上下文）
  - 讨论时长（分钟）
  - 每组人数（建议 4-6 人）
- 查看历史 Session 列表

#### 2.3.2 分组管理
- 查看当前已加入等待室的学生列表（含 E/I 类型）
- 点击「开始分组」按钮，系统自动按 E/I 均衡原则分组
  - 分组算法：每组尽量保持 E/I 比例均衡（如 2E2I 或 3E2I）
- 教师可手动调整分组结果
- 确认后广播分组信息，所有学生进入讨论室

> **[难点 ⭐⭐⭐]** E/I 均衡分组算法需处理人数不整除、E/I 比例极端不均等边界情况。广播分组信息需确保所有在线学生均收到（需要消息确认机制）。

#### 2.3.3 实时 Dashboard
- 实时查看各组讨论状态：
  - 每组当前发言人
  - 各组参与度热力图（谁说得多谁说得少）
  - 全场沉默预警（某组长时间无人发言时高亮提示）
- 数据图表：
  - 各组发言时长分布柱状图
  - 个人参与度对比折线图

> **[难点 ⭐⭐⭐]** 教师端需同时订阅所有组的实时数据流，数据量随组数线性增长。图表需要实时刷新且不能因频繁重渲染导致页面卡顿，需做数据聚合和渲染节流。

#### 2.3.4 讨论结束
- 教师可手动结束讨论或等待计时结束
- 结束后展示全场数据总览
- 支持导出数据（CSV）

## 3. 非功能需求

| 项目 | 要求 |
|---|---|
| 实时延迟 | 发言状态同步延迟 < 500ms | > **[难点]** 依赖 WebSocket 连接质量，弱网环境下需有降级方案（轮询兜底） |
| 并发 | 支持单 Session 最多 60 名学生同时在线 | > **[难点]** 60路实时消息广播，需评估 Supabase 免费套餐连接数上限（免费版限 200 并发连接） |
| 响应式 | 适配移动端（手机竖屏）和 PC 端 |
| 浏览器兼容 | Chrome / Safari 最新版 |
| 可用性 | Live URL 在评分期间持续在线 |

## 4. 技术方案

### 4.1 推荐方案 A（快速）
| 层级 | 技术 |
|---|---|
| 前端 | React + TypeScript + TailwindCSS + Framer Motion |
| 后端/DB | Supabase（PostgreSQL + Realtime + Auth） |
| AI | DeepSeek API |
| 部署 | Vercel（前端）+ Supabase Cloud |

### 4.2 备选方案 B（熟悉 Java）
| 层级 | 技术 |
|---|---|
| 前端 | React + TypeScript + TailwindCSS + Framer Motion |
| 后端 | Spring Boot + WebSocket（STOMP） |
| 数据库 | MySQL |
| AI | DeepSeek API |
| 部署 | 自有服务器（Nginx + Java） |

## 5. 数据库核心结构

```
users           → id, email, name, role, personality(E/I), created_at
sessions        → id, topic, duration, status, teacher_id, created_at
groups          → id, session_id, name
group_members   → id, group_id, user_id
speaking_logs   → id, user_id, group_id, start_time, end_time, duration
scores          → id, user_id, group_id, session_id, total_score
ai_prompts      → id, group_id, session_id, trigger_type, content, created_at
```
